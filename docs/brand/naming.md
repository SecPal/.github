# Naming

Use these names exactly in public and contributor-facing materials.

## Required Names

| Context                                            | Use                    | Do not use                                                  |
| -------------------------------------------------- | ---------------------- | ----------------------------------------------------------- |
| Platform, suite, organization-level product family | `SecPal`               | `Secpal`, `SecPAL`, `SecPal Suite` as a formal name         |
| Standalone product, first public mention           | `GuardGuide by SecPal` | `SecPal GuardGuide`, `Guard Guide`, `GuardGuide for SecPal` |
| Standalone product, later nearby references        | `GuardGuide`           | `GG`, `the guide app`                                       |

## Public Copy Rules

- Use `SecPal` for the platform or suite.
- Use `GuardGuide by SecPal` for the standalone product within the SecPal family.
- After the full product name appears once in the same document or screen, `GuardGuide` is acceptable where the SecPal relationship is already clear.
- Keep repository names, package names, domains, and code identifiers in their technical form when documenting commands or file paths.
- Do not invent campaign names, editions, or abbreviations unless a brand document adds them explicitly.

## Rationale

The product family needs one stable umbrella name, and each standalone product needs a public name that does not look like an internal component. The first-mention rule gives readers the full relationship once, then avoids repetitive copy.
